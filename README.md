# Embeded-edge-aware-scheduler
Repo of the Embeded-edge-aware-scheduler project between FH-Dortmund and Universidad del Azuay


## INTRODUCTION
The proliferation of IoT and the success of cloud services have pushed the horizon of a new computing paradigm, edge computing. This emerging technology places substantial processing and storage resources closer to the user equipment (UE), minimizing response time and improving quality of service. The applications that benefit from any of the attributes unique to edge computing are referred to as edge-dependent applications. In order to take full advantage of the edge and to offer a broad service coverage, a wide range of real-time edge-dependent applications use the mobile network to establish the connection with the edge server node. This is better known as Mobile Edge Computing (MEC). From the UE perspective, the usage of MEC introduces several challenges to be solved. One of those is the scheduling of edge-dependent tasks. In real time systems, tasks must be accomplished before the imposed deadline to avoid QoS degradation or even system failure. Current real-time schedulers do not contemplate factors inherent to mobile edge computing such as: resource allocation, edge processing time, latency, packet loss, bandwidth, etc. which undoubtedly affect the functionality and performance of the system. This study seeks for the implementation of an embedded edge-aware scheduler on the user equipment side, which is in charge of planning local and edge-dependent tasks execution, regardless of their primary execution site (local or edge), in order to optimize system and infrastructure performance and thus, improve QoS. 

## PROBLEMATIC
There are several studies and research carried out in the field of edge resource allocation and edge task scheduling. However, the majority of them tackle this problem from the infrastructure point of view, and therefore, we know very little about the real influence of executing edge-dependent tasks from the UE perspective, especially when several tasks with real-time requirements run in the same UE, which is a very accurate assumption.
There must be a seamless communication between the UE and the edge infrastructure, that is to say, both elements must be aware of the working conditions of its counterpart. Normally, it is expected and supported that the infrastructure is able to know application requirements prior to connection. In general terms, the UE is who decides whether to connect or not to a certain edge node based on multiple metrics. Previous studies have introduced end-to-end latency as an accurate metric to infer a system´s QoS. Nevertheless, to the best of the knowledge of the authors of this project, until now there is no study that considers the realistic fact that a UE could execute more tasks than only one edge-dependent task. When this condition takes place, the main challenge MEC will face is that the E2EL perceived by UE will not only affect the edge-dependent task execution time. In reality, this fact could create a waterfall effect, where other tasks running in the same device (independent of their primary execution site) could be affected as well, as processing times could be shifted dynamically and thus system predictability is harder to reach. Contrarily, the continuous execution of embedded local tasks could also prevent an edge-dependent task from reaching its deadline, resulting in QoS degradation. The main purpose of a real-time operating system is to reach a so-called predictable state, however as explained above, the incorporation of edge-dependent tasks to these systems represents a big challenge due to the unpredictable nature of MEC. This could be improved if edge resources allocated to an edge-dependent task are not only assigned based on the unique task requirements, but instead, based on the requirements of the complete system running on the UE as a single unit.
